Why Userflow?

March 5, 2024 Â· 3 min read
Matt Dupree
Maintainer of Docusaurus

I once logged into a SaaS app and was hit with 2 different modals announcing new features and welcoming me into the app. Later that day, I was sent a useless email explaining how to use a feature that I had already learned how to use after I logged in. Unfortunately, this happened on a SaaS app that my team was building.

Fixing this bad experience wasn't easy. Our onboarding code was:

  1. Spread out all over the codebase like spaghetti.
  2. Hardcoded into clients.
  3. Tightly coupled with vendor code and SaaS tools.

We all knew we shouldn't write hardcoded, tightly-coupled, spaghetti code, and yet, over time, that's how our onboarding code evovled. Different engineers working on different projects over time didn't see the mess they were going to make. A little hardcoded product tour built with react-joyride seemed harmless. So did tightly coupling our email sending logic with Sendgrid's SDK.

This sort of thing had happened before, and upon reflection this isn't suprising. Onboarding code often isn't fun to write, and product and marketing are often in a hurry to get it done. To work on these sorts of projects without guardrails is like writing C++. In both cases, you're circling the pit of despair:

That's the problem with C++. It does a terrible job of protecting you from your own worst enemy â yourself. When you write code in C++, you're always circling the pit of despair, just one misstep away from plunging to your doom.

We wrote Userflow to create the pit of success for onboarding code, to make it easy to write centralized (cohesive), loosely-coupled, server-configurable onboarding experiences across applications, email, and beyond.

The gist of our approach is to create a kind of orchestration engine for these experiences. Something like Airflow, but specifically designed for user onboarding experiences. Userflow gives you a declarative way specify the logic for which user see what at what time. Then it fetches the user data relevant to this logic from multiple sources to ensure that it is executed flawlessly.

The following code would fix the bad onboarding experience I started this post with:

// server-config.ts
const config: Config = {
  user: {
    properties: [
      {
        type: "postgres",
        name: "profileCreatedAt",
      },
      {
        type: "postgres",
        name: "profilePhotoCreatedAt",
      },
    ],
  },
  flows: [
    {
      name: "welcome",
      when: "profileCreatedAt === null",
      steps: [
        {
          type: "banner",
          content: "Welcome to the app! Complete your profile!",
        },
        {
          type: "modal",
          content: "We just shipped profile photos!",
        },
      ],
    },
  ],
};

export default config;
// app.jsx
const App = ({userEmail}) => {
    const { flow } = useUserFlow(userEmail)
    const {name, content} = flow
       return (
        <WelcomeBanner show={name === 'welcome'} content={content}/>
        <AnnounceModal open={name === 'announce'} content={content}>
        <Heading>
            Good apps use userflow.
        </Heading>
        <Button>Ok</Button>
    )
}
